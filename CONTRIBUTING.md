# Руководство для контрибьюторов ZapretGUI

Добро пожаловать в проект ZapretGUI! Мы рады вашей заинтересованности и готовы принять ваш вклад в развитие проекта.

## Как внести вклад

### 1. Выбор задачи
- Проверьте список [issues](https://github.com/your-repo/issues) на наличие подходящих задач
- Выберите задачу, соответствующую вашим навыкам и интересам
- Оставьте комментарий к issue, указав ваше намерение работать над задачей

### 2. Настройка окружения
1. Склонируйте репозиторий:
   ```bash
   git clone https://github.com/your-repo/ZapretGUI.git
   cd ZapretGUI
   ```

2. Создайте ветку для вашей задачи:
   ```bash
   git checkout -b feature/description
   ```

3. Установите зависимости:
   ```bash
   # Для C# проектов
   dotnet restore
   ```

### 3. Разработка
- Следуйте [Руководству по стилю кода](#стиль-кода)
- Пишите чистый, поддерживаемый код
- Добавляйте комментарии и документацию
- Пишите тесты для нового функционала

### 4. Создание Pull Request
1. Закоммитьте изменения:
   ```bash
   git add .
git commit -m "descriptive commit message"
   ```

2. Отправьте изменения на GitHub:
   ```bash
   git push origin feature/description
   ```

3. Создайте Pull Request в ветку `master`

4. Убедитесь, что все проверки (CI/CD) проходят успешно

## Процесс разработки

### Ветвление
- `master` - стабильная версия
- `develop` - текущая разработка
- `feature/*` - новые возможности
- `bugfix/*` - исправления ошибок
- `hotfix/*` - срочные исправления

### Рабочий процесс
1. Выбираем задачу из backlog
2. Создаем feature ветку
3. Разрабатываем и тестируем
4. Создаем Pull Request
5. Проходит code review
6. Мержим в основную ветку

## Требования к коду

### Общие принципы
- Следовать принципам SOLID
- Использовать понятные имена переменных и методов
- Минимизировать сложность кода
- Добавлять документацию к сложным функциям

### C# специфика
- Стиль кода: Microsoft C# Coding Conventions
- Использовать async/await для асинхронных операций
- Минимизировать использование static классов
- Следовать паттерну MVVM

### WPF специфика
- Использовать data binding вместо code-behind
- Следовать принципам MVVM
- Использовать ресурсы и стили
- Оптимизировать производительность UI

### Тестирование
- Покрывать unit тестами критические пути
- Использовать TDD подход при возможности
- Тестировать как положительные, так и отрицательные сценарии

## Процесс PR

### Что должно быть в PR
- Описание изменений и их цели
- Список затронутых файлов
- Инструкции для тестирования
- Скриншоты (если изменяется UI)

### Процесс code review
1. Другой разработчик ревьюит код
2. Обсуждаем замечания и предложения
3. Вносятся правки при необходимости
4. PR принимается и мержится

### Критерии принятия
- Код проходит все тесты
- Соответствует стилю проекта
- Не ломает существующий функционал
- Документация обновлена

## Рекомендации по документации

### Комментарии к коду
- Использовать XML документацию для публичных API
- Объяснять сложную логику
- Документировать неочевидные решения

### Commit сообщения
- Использовать conventional commits:
  - `feat:` - новые возможности
  - `fix:` - исправления ошибок
  - `docs:` - изменения в документации
  - `style:` - форматирование кода
  - `refactor:` - рефакторинг
  - `test:` - добавление тестов

### Pull Request описания
- Краткое описание изменений
- Связь с issue (если есть)
- Инструкции для тестирования
